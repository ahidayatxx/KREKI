# Systematic Literature Review: Task Management System
## Breaking Down the SLR Plan into Manageable Tasks

---

## Document Purpose

This document provides a granular task breakdown system for conducting systematic literature reviews. It enables:

1. **Task Tracking**: Monitor progress through clearly defined tasks
2. **Project Continuity**: Resume work seamlessly across sessions
3. **Status Documentation**: Maintain complete record of completed and pending tasks
4. **Quality Control**: Ensure no steps are missed
5. **Team Coordination**: Assign and track individual responsibilities
6. **Progress Reporting**: Generate status updates for stakeholders

---

## How to Use This Task System

### Task Organization
- **Phases**: Major review stages (Protocol, Search, Selection, etc.)
- **Tasks**: Specific activities within each phase
- **Subtasks**: Detailed steps within each task
- **Status Tracking**: Mark tasks as Not Started, In Progress, Completed, or Blocked

### Project File Naming Convention
```
project-[ReviewTopicShortName]-[YYYY-MM-DD].md

Examples:
project-CBT-vs-Medication-GAD-2025-01-15.md
project-Telemedicine-Diabetes-2025-02-20.md
project-AI-Education-SLR-2025-03-10.md
```

### Status Update Protocol
1. **Daily**: Update task status at end of work session
2. **Weekly**: Create new project file with current status
3. **Monthly**: Archive completed project files
4. **Phase Completion**: Generate phase completion report

### Task Status Codes
- `[ ]` Not Started
- `[~]` In Progress
- `[✓]` Completed
- `[!]` Blocked (specify reason)
- `[?]` Needs Clarification
- `[S]` Skipped (with justification)

## NOTES FOR CLAUDE CODE USAGE

### Task Breakdown Philosophy
This task system breaks the SLR into ~150-200 granular tasks because:

1. **Clarity**: Each task has clear start/end points
2. **Trackability**: Progress is measurable and documentable
3. **Resumability**: Easy to resume work after breaks
4. **Quality**: Ensures no steps are skipped
5. **Accountability**: Clear responsibility assignment

### When to Update Project Files

**Daily Updates** (during active work):
- Update task status codes
- Add notes to active tasks
- Document decisions made
- Log any issues

**Weekly Updates** (create new file):
- Create new `project-[name]-[date].md`
- Update all status codes
- Update progress metrics
- Add weekly summary

**Milestone Updates** (create new file):
- At phase completions
- Major deliverable completion
- Significant decisions
- Timeline changes

### Best Practices

1. **Be Specific in Notes**: Don't just mark complete, note what was actually done
2. **Document Rationale**: Explain why decisions were made
3. **Flag Issues Early**: Mark blockers immediately with clear description
4. **Update Dependencies**: When completing tasks, check what becomes unblocked
5. **Maintain Continuity**: Each project file should tell the story of progress

### Integration with Claude Code

Claude Code should:
1. Always load the most recent project file first
2. Understand current phase and active tasks
3. Suggest next logical steps based on dependencies
4. Help update status codes and notes
5. Generate progress reports when requested
6. Alert to potential issues or blockers
7. Ensure nothing is missed in the systematic process
